# tests/ -- Test Code

> Module-level conventions for Claude Code. For automated validation, use `/validate`.
> For test commands, pass criteria, and diagnostics, see `docs_ai/testing/test_guide.md`.
> For integration test case catalog and e2e test templates, see `.claude/skills/validate/SKILL.md`.

## Test Categories

### Unit Tests (unittest/)
- Framework: CUnit (>= 2.1)
- Entry point: `unittest/main.c` -- registers all test suites
- Naming: `xqc_<module>_test.c/.h` -- each file is a CUnit test suite
- Binary: `run_tests` (built by CMake)
- Run: `./build/run_tests` from project root

### Integration Tests (test_client.c / test_server.c)
- Full QUIC client/server using libevent for I/O
- Driven by `scripts/case_test.sh` which starts server, runs client scenarios, checks results
- Uses command-line flags (`-x <N>`) for test case selection

## Adding a New Unit Test

1. Create `unittest/xqc_<name>_test.c/.h`
2. Define test functions: `void xqc_test_<name>_<case>(void)`
3. Register suite in `unittest/main.c`: `CU_add_suite()` + `CU_add_test()`
4. Add source file to `CMakeLists.txt` under the `run_tests` target

## Conventions

- **Test naming**: Test functions are `xqc_test_<module>_<scenario>`. Suite names match the module being tested.
- **Assertions**: Use CUnit macros: `CU_ASSERT()`, `CU_ASSERT_EQUAL()`, `CU_ASSERT_PTR_NOT_NULL()`, etc.
- **Setup/teardown**: CUnit supports per-suite init/cleanup functions. Use them for allocating/freeing test fixtures.
- **Platform abstraction**: `platform.h` provides cross-platform helpers for test binaries.

## Common Pitfalls

- **TLS certificates**: Integration tests require TLS certificates. Generated by the build/test scripts. Don't hardcode certificate paths.
- **Port conflicts**: `test_server.c` binds to a port. If tests fail with address-in-use, check for lingering processes.
- **Feature-gated tests**: Some tests (FEC, BBR2, etc.) are conditionally compiled. Match `#ifdef` guards to the feature flags in `CMakeLists.txt`.
